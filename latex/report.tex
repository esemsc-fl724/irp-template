\section{Introduction}
\subsection{Problem Description and Significance}
Weather, as the most critical environmental information after time and place, has a direct and far-reaching impact on people's daily lives. However, traditional APIs or weather query systems mostly use template-based responses or keyword matching, which are unable to understand the diversified expressions of user's natural language, and are difficult to adapt to the increasingly complex query needs of users. 

In recent years, with the rapid development of LLMs in natural language processing, more and more users have started to use conversational LLMs like ChatGPT (OpenAI, 2023)~\cite{OpenAI2023} for various queries. However, many user questions require knowledge beyond what a pre-trained LLM can store internally.~\cite{Zhang2023} For example, a user might ask, "What’s the weather like in London right now?" Clearly, such tasks require accessing real-time data through external APIs.~\cite{Chadha2024} The ability to interact with tools is a key step in the evolution of LLMs from passive text generators to action-capable AI agents.

Nevertheless, applying LLMs to real-world weather queries presents several challenges, such as: how to understand the user's intention, how to call the APIs in a structured way to obtain the external data; and how to semantically process the returned data.

Building an AI Weather Assistant System based on Large Language Models (LLMs) not only can enhance the user experience, but also create a practical example for engineering applications for LLMs. These design patterns can be applied to other agent-based cases such as tourism, health and education etc.


\subsection{Review of Existing Work}
In my research, I have discovered several different methods for designing weather assistant systems.

First, there are traditional weather information systems, which usually rely on APIs and mobile applications to provide weather data~\cite{Zhang2023}. Users must manually interpret numerical weather data, which is not user-friendly, especially in varied real-life situations.

Second, some weather chatbots are built using the Rasa framework~\cite{WeatherBot2018}, which use a combination of traditional NLP, predefined rules, and API calls to deliver responses. However, the disadvantage is that it relies on predefined intents and rules, making it difficult to handle complex real-world questions.

In the past two years, AI agents have become more popular. Developers can use system prompts to tell LLMs what tools are available in the system. Some weather systems~\cite{Jeong2024} have explored the integration of LLMs with weather APIs. 

However, there are still some limitations. For example, the tool lists are directly embedded in the LLM prompt, making it hard to scale and maintain. Also, due to the lack of a clear standard, LLM may return incorrect formats, causing retries and extra token usage. Some well-known agents, such as Cline~\cite{Cline2025}, are using the system prompt method.

After recognizing the problems with system prompts, major model providers started to introduce a more structured way of calling tools, known as Function Calling~\cite{Liu2024}~\cite{Gim2024}. Platforms like OpenAI, Claude, and Gemini~\cite{OpenAI2025}~\cite{Claude2025}~\cite{Gemini2025} have all developed their own Function Calling systems. The core idea is to register each tool using a unified JSON Schema format, and provide that to the model separately from the prompt. I have also seen some articles~\cite{Anthonio2024}~\cite{KermaniWeatherAgent2025} on implementing weather chatbot that have already started using Function Calling.

Function Calling has its benefits: developers do not need to write complex prompts, the system can detect format errors and recover, and it also saves tokens by not repeating tool descriptions. However, the main drawback is the lack of a unified standard. Different providers define Function Calling differently, and some models do not natively support this feature, making tool definitions difficult to reuse.

As a result, both System Prompt–based and Function Calling–based methods for defining tools currently coexist in the market.

To address these issues, providers begin to explore some methods for managing tools, such as the Model Context Protocol (MCP)~\cite{Anthropic2024MCP}, which will be discussed in detail in later sections.

\subsection{Objectives and Expected Deliverables}
Therefore, this project aims to build an LLM-based AI weather agent system that can understand natural language queries, retrieve real-time weather data via the OpenWeather API~\cite{OpenWeatherAPI}, and generate informative and coherent replies to provide users with real and accurate weather data and related information.
